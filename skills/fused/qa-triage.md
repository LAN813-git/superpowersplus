# QA-Triage

Interactive QA session with automatic triage integration. Report bugs conversationally → file issues → move through triage states.

## Workflow

```mermaid
graph TD
    A[User reports bug] → B[QA: Clarify & explore]
    B → C{Scope assessment}
    C →|Single issue| D[File issue]
    C →|Breakdown| E[File multiple issues]
    D → F[Auto-triage]
    E → F
    F → G{State assignment}
    G →|needs-info| H[Ask reporter]
    G →|ready-for-agent| I[Generate agent brief]
    G →|ready-for-human| J[Human implementation note]
    G →|wontfix| K[Close with explanation]
    H →|Reporter replies| F
    I → L[AFK agent picks up]
```

---

## Phase 1: QA Session（交互式报告）

### 1.1 聆听并轻量澄清

用户用自己的话描述问题。最多问 2-3 个简短问题：

- 预期行为 vs 实际行为
- 复现步骤（如果不明显）
- 是否一致/间歇性

**不要过度访谈**。描述足够清晰就继续。

### 1.2 后台探索代码库

同时启动后台 Agent（subagent_type=Explore）理解相关区域：

- 学习领域语言（检查 UBIQUITOUS_LANGUAGE.md）
- 理解功能应做什么
- 识别用户行为边界

**目标**：写更好的 issue，但 issue 本身不引用具体文件/行号。

### 1.3 评估范围：单 issue 还是拆分？

**拆分条件**：
- 修复跨多个独立区域
- 明显可分离的关注点
- 多个不同故障模式

**保持单 issue**：
- 一个行为在一个地方出错
- 所有症状由同一根行为引起

### 1.4 文件化 issue

用 `gh issue create` 创建 issue。**不要让用户先审核**——直接创建并分享 URL。

#### 单 issue 模板

```markdown
## What happened

[用简单语言描述用户遇到的实际行为]

## What I expected

[描述预期行为]

## Steps to reproduce

1. [具体的编号步骤，开发者可跟随]
2. [使用代码库的领域术语，不是内部模块名]
3. [包含相关输入、标志或配置]

## Additional context

[任何额外观察——用领域语言，不引用文件]
```

#### 拆分 issue 模板

按依赖顺序创建（阻塞项优先），以便引用真实 issue 编号。

```markdown
## Parent issue

#<parent-issue-number> 或 "Reported during QA session"

## What's wrong

[描述这个特定行为问题——只是这个切片]

## What I expected

[这个切片的预期行为]

## Steps to reproduce

1. [针对 THIS issue 的步骤]

## Blocked by

- #<issue-number>（如果此 issue 在另一个解决前无法修复）

或 "None — can start immediately"（如果无阻塞）

## Additional context

[与此切片相关的额外观察]
```

**拆分规则**：
- 优先多个薄 issue 而非少数厚 issue
- 诚实标记阻塞关系
- 按依赖顺序创建
- 最大化并行性

#### 所有 issue 正文的规则

- **无文件路径/行号**——会过时
- **使用项目领域语言**
- **描述行为，不是代码**
- **复现步骤必须有**
- **简洁**——30 秒可读完

创建后打印所有 issue URL（总结阻塞关系），问："下一个 issue，还是结束？"

### 1.5 继续会话

持续到用户说结束。每个 issue 独立——不批量。

## Phase 2: Triage（状态机分诊）

Issue 创建后自动进入分诊流程。

### 2.1 角色体系

**类别角色**（2 个）：
- `bug` — 某个东西坏了
- `enhancement` — 新功能或改进

**状态角色**（5 个）：
- `needs-triage` — 维护者需要评估
- `needs-info` — 等待报告者更多信息
- `ready-for-agent` — 完全指定，AFK agent 可用
- `ready-for-human` — 需要人类实现
- `wontfix` — 不会处理

每个分诊的 issue 应恰好携带一个类别角色和一个状态角色。

### 2.2 状态转换

```mermaid
graph LR
    A[unlabeled] → B[needs-triage]
    B → C[needs-info]
    B → D[ready-for-agent]
    B → E[ready-for-human]
    B → F[wontfix]
    C →|报告者回复| B
```

维护者可随时覆盖——标记异常转换并询问。

### 2.3 自动分诊流程

QA 阶段结束后，对每个创建的 issue：

1. **收集上下文**：读取完整 issue（正文、评论、标签、报告者、日期）
2. **探索代码库**：使用项目领域语言，尊重 ADR
3. **检查 .out-of-scope/**：查找先前拒绝的类似 issue
4. **推荐**：给出类别和状态推荐及推理
5. **等待方向**：维护者确认后执行

### 2.4 状态执行

#### ready-for-agent

发布 agent brief：

```markdown
> *This was generated by AI during triage.*

## Agent Brief

### Context
[issue 的简要背景]

### Task
[具体要做什么]

### Acceptance Criteria
- [ ] 标准 1
- [ ] 标准 2

### Technical Notes
[代码库探索的发现，用领域语言]
```

#### ready-for-human

同样结构，但标注为什么不能委托（判断调用、外部访问、设计决策、手动测试）。

#### needs-info

发布分诊笔记：

```markdown
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

#### wontfix（bug）

礼貌解释，然后关闭。

#### wontfix（enhancement）

写入 `.out-of-scope/`，从评论链接到它，然后关闭。

## Phase 3: 恢复与监控

### 3.1 恢复先前会话

如果 issue 上有先前分诊笔记：
- 读取它们
- 检查报告者是否回答了未解决问题
- 呈现更新的图片
- **不重新问已解决的问题**

### 3.2 显示需要关注的内容

查询 issue tracker 并呈现三个桶（最旧优先）：

1. **未标记** — 从未分诊
2. **needs-triage** — 评估进行中
3. **needs-info 且报告者有活动** — 需要重新评估

显示计数和每 issue 一行摘要。让维护者选择。

## 通用规则

### 免责声明

每个在分诊期间发布的评论或 issue **必须**以此开头：

```
> *This was generated by AI during triage.*
```

### 领域语言

- 使用项目的 UBIQUITOUS_LANGUAGE.md（如果存在）
- 不引用具体文件路径或行号
- 描述行为，不是代码

### 快速状态覆盖

维护者说"把 #42 移到 ready-for-agent"时：
- 信任他们并直接应用角色
- 确认你将做什么（角色变更、评论、关闭）
- 然后执行
- 跳过询问

### Issue 持久性

- issue 应在重大重构后仍有意义
- 从用户视角写
- 复现步骤必须有
